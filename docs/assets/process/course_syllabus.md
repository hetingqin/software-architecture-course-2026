## 一、三维审阅：现场落地 × 学生理解 × 内容不过时

第四次调整版总体方向合理，但从三个实战角度再过一遍，仍发现若干需改进之处，记录如下，作为本版修订的依据。

---

### 维度一：现场讲授能否顺畅落地

**问题1 — 时间结构过于刚性**  
"15+90+35+20分钟"的切分在真实课堂中难以执行。学生讨论热起来时不该强制切断，讲师状态好时也不必卡秒表。应改为"弹性节奏"描述，保留几个必须到位的节点即可。

**问题2 — 互动活动缺操作细节**  
"风险工作坊"、"架构评审会"、"故障30分钟应急推演"听起来丰富，但讲师第一次上台时会困惑：几人一组？需要提前发什么材料？没有操作要点，这些活动大概率变成"讲师讲、学生听"。

**问题3 — 第8次课应急推演门槛偏高**  
完整应急推演需要提前设计故障场景、分配角色、准备"日志"道具，160分钟内做完并有反思，对讲师要求很高。建议降为"真实事故案例复盘"，效果不差，门槛低很多。

**问题4 — 第5次课UML覆盖面过宽**  
用例/类/时序/状态/组件/部署六类图全讲，160分钟内只能蜻蜓点水。现代工程中时序图+C4已覆盖绝大多数架构沟通需求，应把C4作为本讲重心，其余UML图降为工具箱参考。

---

### 维度二：学生能否真正理解

**问题5 — 第2次课定位对专硕不够准确**  
研究生本科大多学过基础软件过程，若把第2次课定位成"模型科普"，有经验的学生会觉得在听复习课。正确定位应是"判断训练"：给定真实项目约束，你会选哪种节奏，为什么，错误选择会带来什么代价。

**问题6 — 里程碑"V1"含义不清晰**  
学生看到"需求规格V1""架构方案V1"会困惑：V1是草稿还是正式版？之后还要交V2吗？需要统一定义：V1 = 可在课堂上被其他组指出问题的工作版本。

**问题7 — 架构术语首次出现无铺垫**  
第4次课直接使用 Architectural Drivers，对没有系统架构训练背景的学生是黑盒词汇。建议第1次课即给出"功能需求/质量属性/约束条件"三分法作为全课通用分析框架，后续各讲反复使用，自然内化。

**问题8 — 质量属性停留在概念层**  
学生方案中常见"高可用、高性能、易扩展"三连，毫无约束力。第4次课起应引入"质量属性场景卡"模板，强制训练可度量写法（指标 + 目标值 + 验证方式 + 降级条件）。

---

### 维度三：内容是否经得起2026年的时间检验

**问题9 — AI工具使用后置太明显**  
把AI内容集中在第9次课，隐含了过时假设"AI是新技术，单独讲一讲"。2026年的现实是AI工具已全面融入日常开发，应从第1次课起"穿针引线"式出现，第9次课做工程化系统总结而非启蒙。

**问题10 — 架构图工具停留在手绘/PPT层次**  
2026年主流工程团队用Mermaid / PlantUML / Excalidraw 做架构图，"架构即文档"已是规范实践。第5次课应介绍至少一种工具化表达方式。

**问题11 — 契约测试适用范围未说明**  
契约测试在微服务场景下价值显著，但学生做模块化单体时直接套用会困惑。需在第7次课明确适用条件，避免"学了用不上"的失落感。

## 二、修订目标（针对11个问题）

1. **时间节奏弹性化**：去掉刚性分钟数，改为"四段节奏"锚点
2. **互动活动标准化**：每个互动附"操作要点"，讲师直接按要点组织
3. **第2次课重新定位**：从"模型科普"改为"工程节奏判断训练"
4. **第5次课聚焦C4+时序图**，其余UML图降为工具箱参考
5. **第8次课应急推演改为事故复盘**，降低准备门槛
6. **AI辅助要点全程穿插**，第9次课做工程化系统总结而非启蒙
7. **质量属性场景卡模板从第4次课起全程使用**
8. **产出定义统一**：V1 = 可被同伴评审指出问题的工作版本；期末提交Final版
9. 每次课保持五要素：**核心问题 / 判断框架 / 贯穿案例 / 可提交产出 / 架构收束**

## 四、修订后的10次课专题计划（落地版）

### 通用说明

**课时节奏（弹性四段）**
-  导入锚定（10-20分钟）：提出核心问题，建立与上讲联系
-  主体讲授（80-100分钟）：以框架+案例推进，不追求覆盖所有子知识点
-  课堂演练（20-35分钟）：小组活动或全班讨论，结果进入产出文档
-  收束布置（10-15分钟）：架构影响总结 + 下讲输入说明

**贯穿案例**：实验室预约与设备管理系统（各组可换成自选项目，但课堂演练时须展示自己的案例，不能"空转"）

**产出定义**：V1 = 可在课堂上被其他组指出问题的工作版本，不是完美稿，是可讨论稿。期末提交Final版。

**质量属性场景卡**（第4次课起全程使用）：每条包含指标名称 / 可测量目标值（如P95 < 2s）/ 场景触发条件 / 验证方式 / 可接受的降级策略。

**AI辅助线索**（全程穿针引线，第9次系统化）：每讲标注「AI辅助要点」，明确本讲任务中可以合理使用AI工具的具体场景和边界。

### 第1次课：从"能写代码"到"能控复杂系统"

**核心问题**：同样会写代码的人，为什么在复杂系统里差距天壤之别？
**判断框架**：复杂性四来源（需求变化速率 / 协作规模 / 技术组合深度 / 运行约束）
**讲授重心**：软件工程活动全景地图；架构在工程链条中的位置；**功能需求/质量属性/约束三分法**（全课通用框架，本讲建立，后续每讲复用）；课程项目规则说明
**课堂演练**：个人回顾自己最复杂的一个项目，识别其中哪一类复杂性最关键，2分钟口头分享
> 操作要点：讲师提前在白板写好四类复杂性，边听边标注归类，结束时形成全班分布图，现场点评1-2个典型案例；无需PPT，站起来说即可

**架构收束**：今天建立的三分法是全课分析语言，第4、6、7讲都会用到
**AI辅助要点**：可用AI整理复杂性描述文字，但"判断哪类最关键"需要自己做
**阶段产出**：选题卡（项目名 + 服务对象 + 核心业务场景3条 + 最大复杂性来源判断）

---

### 第2次课：工程节奏设计过程模型判断力训练

**核心问题**：在你的项目约束下，错误的开发节奏会造成什么后果？
**判断框架**：过程模型选择矩阵（需求稳定性  团队规模  交付压力  验证机会频率）
**讲授重心**：不做"5种模型逐一介绍"，而做"判断训练"给定约束，哪种节奏会失败，为什么；瀑布的适用条件与失效条件；迭代与增量的区别（经典混淆点）；敏捷的本质（价值观+实践，不是"没有文档"）；Scrum节奏与研究生项目的对应关系
**课堂演练**：同一个项目（预约系统首版），分别用"瀑布"和"Scrum 2周迭代"设计时间线，然后引入"第3周需求变更"场景，对比两种节奏的应对代价
> 操作要点：2-3人一组，10分钟设计时间线，5分钟全班对比展示，讲师点评2-3组；重点引导学生说出"代价在哪"，而不是"哪种更好"

**架构收束**：过程模型影响架构决策时机瀑布要求架构前置锁定，迭代允许架构渐进演进；各自代价是什么
**AI辅助要点**：可用AI生成迭代计划草稿，但过程模型选择的判断依据（约束分析）需要自己写
**阶段产出**：过程模型选择说明（选了哪种，为什么，放弃了什么，主要风险是什么）+ 迭代计划V1

---

### 第3次课：项目可行性与工程治理

**核心问题**：项目为什么会"技术上可以做，工程上做死"？
**判断框架**：可行性四维（技术/经济/组织/合规） 风险分级矩阵（可能性  影响）
**讲授重心**：可行性四维重点讲组织和合规，这两维学生最容易忽视；范围-进度-成本-质量的真实张力；风险管理不是消除风险，而是识别+分级+制定B计划；布鲁克斯定律的工程意义；团队协作成本与架构复杂度的互放大关系
**课堂演练**：风险识别工作坊每组为自己项目列出8-10个风险，标注可能性和影响，选出Top3制定B计划
> 操作要点：讲师提前准备"风险卡片模板"（共享文档或纸质），格式：风险描述 / 可能性(H/M/L) / 影响(H/M/L) / B计划一句话；各组填写后1-2分钟投影展示，其他组可以补充；选3-4组有代表性的展示即可

**架构收束**：风险清单中的技术风险直接转化为后续架构决策的输入"担心什么，就在架构上预留应对"
**AI辅助要点**：可用AI头脑风暴风险列表，但风险优先级排序和B计划设计需结合自己项目判断
**阶段产出**：可行性摘要 + 风险清单（8条以上，含分级和B计划）+ 项目计划V1

---

### 第4次课：需求工程与架构驱动因素识别

**核心问题**：需求如何从"功能列表"变成"架构决策的真正输入"？
**判断框架**：三分法（功能需求 / 质量属性需求 / 约束条件）+ 质量属性场景卡模板
**讲授重心**：利益相关者分析；场景与用例（不是UML讲解，而是"用什么描述用户真正想要的"）；功能需求与质量属性需求的拆分；**质量属性场景卡**（可度量、有条件、有取舍）本讲核心训练；架构驱动因素 = 影响结构决策的需求+质量属性+约束；需求变更代价：为什么某些变更便宜某些很贵
**课堂演练**：需求变更评审讲师现场宣布"新增需求"（如：系统需要支持微信扫码预约），各组评估对架构影响大小及理由
> 操作要点：变更卡提前准备（1句话描述需求变更），每组5分钟讨论，举手表态"架构影响大/小"，各请一组说理由；讲师引导区分"功能影响"和"架构影响"的差异

**架构收束**：今天的质量属性场景表将在第6次课直接用于检验架构方案的合理性
**AI辅助要点**：可用AI辅助生成用例场景描述，但质量属性目标值需要根据项目实际调研设定
**阶段产出**：需求规格V1 + 质量属性场景表（至少5条，每条含可度量目标值）

---

### 第5次课：架构表达从需求到可讨论的结构模型

**核心问题**：如何把脑子里的系统结构，变成团队可以讨论和评审的东西？
**判断框架**：C4模型（C1-C2-C3层级）+ 时序图
**讲授重心（按重要性排列）**：
-  **C4模型**（C1上下文图/C2容器图/C3组件图）本讲真正重心，讲透
-  **时序图**描述核心场景动态行为，第二重心
-  **现代工具简介**：Mermaid / PlantUML（文本生成图）/ Excalidraw（协作草图）至少现场演示一种
-  **UML其他图**（类图/状态图/部署图）工具箱简介，"知道什么时候用，不必逐一精通"

**课堂演练**：小组共建贯穿案例的C2容器图 + 预约提交核心流程时序图，用任意工具（手绘/Mermaid/在线工具均可）
> 操作要点：明确要求C2层级（主要可部署单元，不是类级别）；互评标准事先写在黑板：1.边界清晰吗？2.依赖方向合理吗？3.有没有遗漏的外部依赖？

**架构收束**：今天的模型图是第6次课架构评审的基础材料，课后完善并提交
**AI辅助要点**：可用AI根据需求描述生成Mermaid代码草稿，但模型正确性（边界是否清晰、依赖是否合理）需要自己审查
**阶段产出**：模型包V1（C1+C2必须；C3关键场景至少一张；核心流程时序图至少一张）

---

### 第6次课：架构设计与风格选型（已有高质量讲稿v3）

**核心问题**：架构决策在哪里影响最大？如何做出可以被审查的决策？
**判断框架**：质量属性权衡矩阵 + ADR写法
**讲授重心**（本讲讲稿已完成v3版）：质量属性权衡；架构风格判断（分层/模块化单体/微服务/事件驱动）；CAP定理与一致性边界划分；ADR写法（重点：放弃方案一栏）
**课堂演练**：架构方案辩论给出两个方案（模块化单体 vs 微服务首版），在给定约束下各组举证支持哪个
> 操作要点：提前准备"约束卡片"（团队5人 / 6周工期 / 机房部署 / 无专职运维），打印后贴在黑板；辩论时必须援引约束卡片，不能只说"微服务更先进"；每方陈述3分钟，讲师做裁判并说明判断标准

**架构收束**：ADR是本讲最重要的产出，要求每组课堂上起草至少1条，课后补齐到3条
**AI辅助要点**：可用AI汇总不同架构方案优缺点，但在本项目约束下的权衡判断必须自己写
**阶段产出**：架构方案V1（含C2图更新版）+ ADR（至少3条，每条包含"放弃方案"说明）

---

### 第7次课：测试策略与质量保障体系

**核心问题**：质量如何被"设计出来"而不只是"测出来"？
**判断框架**：测试金字塔  架构层对应关系
**讲授重心**：测试金字塔（单元/集成/端到端）与架构的关系**可测试性由架构决定，不是测试工具决定**；质量门禁（Quality Gate）：什么标准下构建才能进入下一阶段；缺陷生命周期与工具；安全基础（认证/越权/注入）与架构权限层的对应；契约测试**适用条件：系统有多个独立部署服务；若是模块化单体，用接口测试替代**（明确说明，避免困惑）
**课堂演练**：为第4次课识别的关键质量属性场景，设计对应测试策略：用哪一层测试覆盖，通过标准是什么，如何自动化
> 操作要点：每组直接在质量属性场景表上补充"测试方案"列，完成后与相邻组交换检查；检查标准：测试层级是否选对？通过标准是否可量化？

**架构收束**：测试难写通常是架构问题记录本组系统中"最难测的地方"，分析是否对应某个架构耦合点
**AI辅助要点**：可用AI生成测试用例骨架，但测试策略选择（用哪层测试、测什么边界）需要自己设计
**阶段产出**：测试计划V1 + 质量门禁清单（包含进入下一迭代的可量化标准）

---

### 第8次课：交付发布、运维与可观测性

**核心问题**：为什么很多系统"开发完成"后反而开始真正出问题？
**判断框架**：可观测性三支柱（日志/指标/链路追踪）+ 发布策略决策树
**讲授重心**：CI/CD流水线基本结构（自动化构建/测试/部署）；发布策略（蓝绿/灰度/金丝雀）的适用场景与回滚机制；可观测性三支柱OpenTelemetry是当前标准；故障复盘方法论（5 Whys / 时间线还原 / 改进项落地）
**课堂演练**：真实事故案例复盘（讲师提前准备公开事故报告，推荐Cloudflare、GitHub等有公开Post-mortem的案例），全班一起还原时间线，找根因，讨论"架构层面的改进是什么"
> 操作要点：讲师打印或投影事故摘要（控制在1页），提前准备3个引导问题（如：故障根因是什么？哪个架构决策放大了影响？如果提前有可观测性，什么时候能发现？）；判断思路才是重点，不需要技术细节完全匹配

**架构收束**：可观测性接入点必须在架构设计时预留本组系统当前是否有日志追踪ID、关键指标埋点？本讲后补充到架构文档
**AI辅助要点**：可用AI辅助生成CI配置文件骨架或日志格式规范，故障复盘中的"为什么"需要人来判断
**阶段产出**：发布运维方案V1（含发布策略说明、回滚方案、关键监控指标清单）

### 第9次课：AI时代的架构工程系统化总结

**核心问题**：AI工具全面介入后，什么变了，什么没变，架构师的价值在哪里？
**判断框架**：AI工具边界图（AI擅长 / AI不擅长 / 必须人工判断的任务清单）
**讲授重心**（本讲是全课AI线索的系统化收束，不是启蒙课）：AI辅助开发边界（+30-50%样板代码效率，但架构判断质量对结果影响反而更大）；Agent架构组成（LLM核心/工具层/记忆层）与Multi-Agent协作；LLM集成四大工程挑战（非确定性/秒级延迟/Token成本/上下文窗口）；RAG架构（向量数据库+检索增强生成）；AI时代架构新原则（可评估性/Prompt版本管理/AI依赖解耦/人工接管通道）；治理与合规（数据安全边界、模型幻觉工程防控）
**课堂演练**：各组为自己的系统设计"AI介入点清单"哪些功能适合引入AI，接入点在哪，会带来哪些新的架构风险，人工接管如何设计
> 操作要点：格式：功能描述 / AI介入方式 / 新增架构风险 / 人工接管触发条件；各组展示1-2条，讲师帮助识别常见误区（如"什么都想用AI做"或"完全不知道怎么用"）

**架构收束**：回顾全课10讲的架构判断力训练路径从第1次的复杂性识别，到现在的AI治理，是同一套框架的层层展开
**AI辅助要点**：全课AI使用情况总结记录最有效的3次AI使用和最"掉坑"的1次，这本身是方案的一部分
**阶段产出**：AI使用规范 + 架构增补ADR（针对AI集成点新增至少1条）+ 全课AI使用记录

### 第10次课：综合评审与项目答辩

**核心问题**：你的方案能否被不了解你们项目的人"一眼看懂，立刻指出问题"？
**形式**：项目答辩（各组15分钟陈述 + 10分钟交叉评审）
**陈述结构要求**： 问题定义（2分钟） 架构决策及依据（5分钟，核心ADR） 质量属性验证方案（3分钟） 演进路径与技术债（3分钟） 最大风险与应对（2分钟）
**交叉评审规则**：每组至少向另一组提1个"可量化"的质疑（如：你们说高可用，指标是什么，如何验证？）
**课程总结**：讲师做全课主线复盘，10讲是一条完整的工程判断力训练链
**最终产出**：架构文档包（需求规格Final + 模型包Final + 架构方案Final + ADR全集 + 测试计划Final + 发布运维方案Final + AI使用规范Final）

## 七、讲师备课要点

**每次课结束前自查6项：**
1. 核心问题是否被正面回答（不是"大家今天应该有收获了"式结尾）
2. 是否完成至少1次"给定约束  推导决策"的推演，而非只讲结论
3. 质量属性是否出现过可测量目标值（数字，不是"高"、"好"）
4. 课堂演练结果是否真正进入了产出文档
5. 架构收束：本讲改变了系统设计的哪一处
6. 下讲输入是否具体说明（不是"下次讲测试，提前预习"）

**互动活动通用操作规则：**
- 分组：2-4人，全课固定分组，不要每次重分
- 时间控制：活动开始前公布结束时间，时间到收结论
- 呈现：每次活动至少有1组展示，讲师当场评价（表扬具体点 + 指出1个问题）
- 结果留存：填入共享文档或拍照保留，纳入平时分评估依据
