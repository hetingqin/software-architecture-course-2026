# 第4章 需求工程：从用户声音到可验证规格

## 4.1 本章导读：需求不是用户说的一句话

需求工程的难点在于，用户表达的往往是愿望、抱怨或解决方案，而不是经过分析的真实需求。软件工程师需要把这些声音转化为目标用户、使用场景、真实问题、系统边界和可验证规格。

实验室预约系统中，学生可能说“希望预约更快”，教师可能说“希望安排更清楚”，实验员可能说“希望少沟通”，学院管理者可能说“希望设备利用率可统计”。这些话都重要，但它们不能直接变成系统设计。需要继续追问：谁在什么情况下使用？当前痛点是什么？成功标准是什么？哪些规则必须满足？哪些信息必须记录？哪些质量属性不可妥协？

本章从利益相关者、需求获取、典型用户、用例分析、规格说明和原型设计几个方面，说明如何把模糊声音变成可开发、可测试、可评审的需求材料。

【你要记住】需求分析不是收集功能清单，而是把用户声音转化为可验证的系统约定。

## 4.2 一个反面案例：“更快的马车”式需求

假设学生反馈：“预约设备太麻烦，希望做一个更快的预约按钮。”如果项目组直接理解为“在首页放一个一键预约按钮”，可能很快做出功能。但试用后发现问题更多：学生不知道哪台设备可用，教师不清楚申请理由，实验员仍然要反复确认冲突，系统还可能自动提交错误预约。

“更快的预约”只是表层表达。继续追问后，真实需求可能包括：学生希望少填写重复信息；系统能提前过滤不可用设备；申请理由模板化；冲突在提交前提示；预约状态及时反馈。也可能发现不同用户的“快”含义不同：学生关注提交快，实验员关注审批清楚，教师关注安排可见，管理员关注规则可控。

需求工程的价值就在这里：不把用户的第一句话当作最终答案，而是通过场景、用户、问题和规则分析出真正需要解决的系统问题。

## 4.3 利益相关者：先识别谁会受影响

利益相关者不仅包括直接使用系统的人，也包括受系统结果影响、维护系统、监管系统或为系统提供外部能力的人。

在实验室预约系统中，学生是直接用户，教师或导师关注实验安排，实验员负责设备开放和审批，学院管理员关注统计和规则，信息中心关注统一认证和安全合规，开发团队关注实现成本和维护边界。若只听学生需求，就可能忽略审批规则、设备维护和数据安全；若只听管理者需求，又可能做出学生难以使用的系统。

利益相关者分析通常要记录：角色、目标、痛点、关心指标、可能冲突和决策权。需求不是投票结果，而是在多个利益之间寻找可实现的平衡。

## 4.4 需求获取：问问题要问到场景里

需求获取可以采用访谈、问卷、观察、文档分析、原型试用和工作坊等方式。关键不是方法名称，而是能否获得真实场景。

访谈时应避免只问“你想要什么功能”，而应问：你现在如何预约？最麻烦的一步是什么？发生冲突时怎么处理？哪些信息必须审批人看到？哪些情况必须拒绝？系统出错时你希望看到什么提示？

观察能发现用户没有说出的流程。例如，实验员可能口头说“审批很简单”，但实际会先看设备状态、实验课程安排、维护计划、申请理由和导师信息。文档分析能发现制度要求，例如开放时间、借用规则、责任归属和安全要求。原型试用能帮助用户发现自己真正关心的交互细节。

【易错点】问卷适合收集趋势，不适合单独定义复杂流程。关键业务规则应通过访谈、观察和原型反复确认。

## 4.5 典型用户与用户故事

典型用户（Persona）用于描述一类用户的目标、行为和痛点。它不是虚构人物传记，而是帮助团队保持需求判断的一致参照。

例如，“研一学生小李”需要快速查看设备空闲时间，提交预约并及时知道结果；“实验员王老师”需要审批清楚、减少重复沟通、避免设备冲突；“学院管理员”需要统计设备利用率并设置规则。

用户故事通常写成：“作为某类用户，我希望完成某个目标，以便获得某种价值。”例如：“作为研究生，我希望在预约前看到设备可用时间段，以便避免提交无效申请。”用户故事后面必须配验收条件，否则容易停留在口号。

【项目提示】每个核心用户故事至少写3条验收条件。例如“显示可用时间段”应说明数据来源、冲突规则、刷新时机和异常提示。

## 4.6 用例分析：把交互过程说完整

用例描述用户与系统之间为达成目标而发生的一组交互。一个好的用例不仅写主成功场景，还要写前置条件、后置结果、异常流程和业务规则。

以“提交实验室预约申请”为例。前置条件是用户已登录且具备预约权限；主成功场景包括选择实验室或设备、选择时间段、填写用途、系统校验冲突、提交申请、生成待审批记录、发送通知；异常流程包括时间段不可用、设备维护中、用户权限不足、网络或通知失败；后置结果是预约记录进入待审批状态，并可被审批人查看。

这种写法比“用户提交预约”更可开发，也更可测试。开发人员知道要实现哪些接口，测试人员知道要验证哪些异常，设计人员知道状态如何变化。

| 主成功场景 | 学生选择设备和时间段；填写用途和导师信息；系统校验权限、设备状态和时间冲突；系统生成待审批记录；系统通知审批人 |

| 后置结果 | 预约进入待审批状态，并可在学生端和审批端查询 |
| 业务规则 | 同一设备同一时段只能有一条有效预约；取消后释放时间段；审批操作必须记录操作者和时间 |

【项目提示】用例文本要能支撑开发和测试。若异常流程写不出来，通常说明需求还没有分析清楚。

## 4.7 需求规格说明：把需求写到可验证

需求规格说明书（SRS）不是越厚越好，而是要把关键需求写清楚、写一致、写到可验证。常见内容包括：项目背景、目标用户、系统范围、功能需求、数据需求、质量属性、约束条件、接口需求、验收标准和术语表。

可验证是需求规格的核心。模糊需求如“系统响应要快”无法直接测试；可验证写法应改成“预约列表在1000条记录以内，常规查询响应时间不超过2秒”。“权限要安全”也过于笼统，可写成“学生不得访问审批接口；未授权访问返回403；所有审批操作记录操作者、时间和结果”。

需求规格还要保持一致性。若用例中写“取消预约后释放时间段”，状态机和数据库字段也应支持取消状态；若质量属性要求“预约不能重复”，并发测试和数据库约束必须体现。

**可验证规格示例**

| 模糊写法 | 可验证写法 |
|---|---|
| 系统不能重复预约 | 对同一设备、同一时间段，数据库中最多只能存在一条状态为待审批或已通过的预约记录；并发提交时只有一个请求成功 |
| 权限要安全 | 学生不得访问审批接口；未授权请求返回403；所有审批操作记录操作者、时间、结果和IP |
| 通知要可靠 | 预约状态变化后生成通知任务；通知失败不影响预约状态；失败任务记录原因并允许重试 |

【考试提示】“可验证需求”通常要写出条件、响应和度量。只写“快、稳定、安全、好用”不能作为合格答案。

## 4.8 原型设计：用低成本暴露理解偏差

原型设计适用于流程、界面和交互不确定的场景。低保真原型可以快速讨论布局和流程，高保真原型可以验证交互细节，但不应过早追求视觉完整。

实验室预约系统的原型应重点展示四个页面：设备列表、可用时间选择、预约申请表、审批详情。原型评审时要观察用户是否能找到可用设备，是否理解状态含义，是否知道失败原因，审批人是否能判断申请是否合理。

原型反馈要转化为需求修改，而不是停留在“用户觉得还可以”。例如，若用户多次误解“待确认”和“待审批”，就应调整状态命名；若审批人反复要求查看导师信息，就应把该字段纳入申请详情。

**原型反馈前后对照**

| 用户反馈 | 需求修订 |
|---|---|
| 学生看不懂“待确认”和“待审批”的区别 | 统一状态命名为“待审批”，并在详情页显示下一步处理人 |
| 审批人希望看到导师和实验用途 | 预约申请表增加导师、用途、预计人数，并在审批详情中展示 |
| 实验员担心设备维护时仍被预约 | 设备状态增加“维护中”，维护中设备不进入可预约时间段 |
| 学生不知道通知失败是否影响预约 | 预约结果页明确显示预约状态，通知失败只作为提示，不改变预约状态 |

## 4.9 质量属性需求：把“好用、安全、稳定”写具体

质量属性需求经常被写得过于抽象。应使用场景化方式描述：刺激源、刺激、环境、制品、响应和响应度量。

例如，安全性场景可以写为：普通学生在已登录环境下尝试访问审批接口，系统应拒绝访问并返回403，同时记录安全日志。可用性场景可以写为：通知服务短时不可用时，预约记录仍应成功保存，通知进入重试队列。性能场景可以写为：学生查询一周内可预约时间段，常规数据量下2秒内返回。

这种写法让质量属性进入设计和测试，而不是停留在“系统要稳定”的口号。

## 4.10 项目实践：写一份轻量需求包

课程项目建议提交一份轻量需求包，至少包括以下内容。

| 模块 | 内容 |
|---|---|
| 利益相关者表 | 角色、诉求、痛点、冲突 |
| 典型用户 | 2-3类核心用户画像 |
| 用例文本 | 3个核心用例，含异常流程 |
| 需求规格 | 功能、数据、质量属性、约束 |
| 原型说明 | 关键页面和反馈记录 |
| 验收标准 | 每个核心需求如何验证 |

这份材料不追求篇幅，而追求可用于开发、测试和答辩说明。

本章到下一章的衔接：需求规格说明了系统要解决什么问题，但还需要图示把对象、流程、边界和结构表达出来。第5章将把这些需求转化为系统建模图。

## 4.11 本章小结

1. 需求不是用户第一句话，而是经过场景、用户、问题和规则分析后的系统约定。
2. 利益相关者分析能避免需求只满足单一视角。
3. 用例分析要写主流程、异常流程、前置条件、后置结果和业务规则。
4. 需求规格必须可验证，质量属性也应写成可测试场景。
5. 原型设计用于暴露理解偏差，并推动需求修订。

## 4.12 复习题与参考答案要点

**基础概念题1：什么是利益相关者？**
参考答案要点：直接或间接受系统影响、参与使用、维护、监管或提供外部能力的人或组织。

**基础概念题2：用例描述应包含哪些要素？**
参考答案要点：参与者、目标、前置条件、主成功场景、异常流程、后置结果和业务规则。

**基础概念题3：什么是可验证需求？**
参考答案要点：能通过评审、测试或检查判断是否满足的需求，通常包含明确条件和度量标准。

**简答题1：为什么不能把用户说的“更快”直接当作需求？**
参考答案要点：更快可能指提交步骤少、反馈快、审批快、冲突少等不同目标，需要通过场景和问题分析澄清。

**简答题2：说明原型设计在需求分析中的作用和风险。**
参考答案要点：作用是低成本暴露理解偏差、验证流程和交互；风险是被误当成正式系统或只关注界面不关注规则。

**案例分析题：某预约系统需求写为“权限要安全”。请改写为可验证需求。**
参考答案要点：按角色限制接口访问；未授权返回403；审批操作记录日志；越权访问纳入测试用例；权限变更有审计记录。

**项目实践题：为你的课程项目写一个核心用例文本。**
参考答案要点：包含参与者、目标、前置条件、主流程、异常流程、后置结果和验收标准。
