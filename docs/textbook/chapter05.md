# 第5章 架构表达与系统建模

## 5.1 本章导读：图不是装饰，而是沟通结构

软件项目中的图示很容易走向两个极端：一种是没有图，所有设计都靠口头说明；另一种是画了很多图，但图只是在摆符号，无法解释系统边界、数据流、职责和约束。真正有价值的建模，是让不同角色能够围绕同一张图讨论问题。

实验室预约与设备管理系统至少需要回答几类问题：系统和校园环境的边界在哪里？有哪些外部系统？内部模块如何划分？核心业务对象是什么？预约从提交到审批经历哪些状态？并发冲突和通知失败在哪个环节处理？

本章介绍 C4 模型、UML 图、ERD、DFD 和流程建模的基本用途。重点不在于背图形符号，而在于选择合适图示表达合适问题，并让图示服务于需求、设计、开发和测试。

【你要记住】好图不是“画得复杂”，而是让边界、关系、职责和流动更清楚。

## 5.2 一个反面案例：图很多，但没人能按图开发

某项目组在设计文档中放了多张图：一个系统框图、一个数据库图、几个流程图和若干页面截图。看起来材料很完整，但评审时仍然回答不出关键问题：统一认证属于系统内部还是外部？预约冲突由前端、后端还是数据库处理？通知失败是否回滚预约？教师审批和实验员审批是同一流程还是两个流程？设备维护状态如何影响可预约状态？

问题不在于没有图，而在于图没有承担明确表达任务。系统框图没有边界，流程图没有角色泳道，数据库图没有说明业务规则，页面截图没有对应需求。图示变成了文档装饰。

改进方式是先确定每张图回答的问题。例如，C1 图回答系统处于什么环境；C2 图回答系统由哪些容器组成；C3 图回答后端内部模块职责；用例图回答参与者和目标；类图回答核心对象关系；状态图回答预约状态变化；DFD 回答数据如何流动。

## 5.3 建模的基本原则：对象、关系、边界、变化

任何建模活动都应关注四个方面。

对象是系统中被讨论的实体，例如学生、设备、预约、审批、通知、实验室。关系说明对象之间如何连接，例如一个学生可以提交多条预约，一台设备在同一时段只能有一个有效预约。边界说明系统内部和外部的分界，例如统一认证和邮件服务通常是外部依赖。变化说明状态如何转换，例如预约从待审批到已通过、已拒绝、已取消。

若图示无法表达这四类信息中的至少一类，就要重新判断它是否必要。

### 5.3.1 从预约用例推导一组图

建模不是先选图形符号，而是从需求问题逐步推导表达方式。以“提交预约申请”为例，可以形成一条最小建模链：

| 推导步骤 | 要回答的问题 | 推荐图示 | 实验室预约系统示例 |
|---|---|---|---|
| 用例分析 | 谁使用系统，目标是什么 | 用例图/用例文本 | 学生提交预约，审批人处理申请 |
| 领域对象 | 系统中有哪些核心概念 | 类图/领域对象表 | 学生、设备、预约、审批记录、通知任务 |
| 数据关系 | 数据如何存储和约束 | ERD | 一台设备同一时段只能有一条有效预约 |
| 状态变化 | 核心对象如何变化 | 状态图 | 草稿、待审批、已通过、已拒绝、已取消 |
| 协作流程 | 多个角色如何完成业务 | 泳道图/活动图 | 学生提交、系统校验、教师审批、系统通知 |
| 系统结构 | 系统边界和模块职责 | C4 C1/C2/C3 | Web、API、数据库、认证、通知服务 |

这条链路能避免“为了画图而画图”。每一张图都回答一个问题，并且能被后续开发、测试或评审使用。

**提交预约申请的最小图例说明**

- 用例图中至少包含学生、审批人、通知服务三个参与者，以及查看可用时段、提交预约、审批申请、发送通知四个用例。
- ERD 中至少包含 User、Device、Reservation、ApprovalRecord、NotificationTask 五个实体，并标明 Reservation 与 Device、User 的关系。
- 状态图中至少包含待审批、已通过、已拒绝、已取消四个状态，并标明“审批通过”“审批拒绝”“用户取消”等触发事件。
- C2 图中至少包含 Web 前端、后端 API、数据库、统一认证、邮件/通知服务。

【项目提示】课程项目的图不要求多，但每张图必须能回答一个具体问题。无法说明用途的图，应删除或重画。

## 5.4 C4 模型：从系统环境到代码组件

C4 模型用四个层次表达软件结构：Context、Container、Component、Code。课程项目通常重点使用前三层。

C1 系统上下文图说明系统与用户、外部系统的关系。实验室预约系统的外部对象包括学生、教师、实验员、学院管理员、统一认证、邮件服务和可能的门禁系统。

C2 容器图说明系统由哪些可运行或可部署单元组成，例如 Web 前端、后端 API、数据库、缓存、文件存储、消息队列和外部服务。C2 图回答的是“系统运行时由哪些部分协作”。

C3 组件图说明某个容器内部如何划分职责。例如后端 API 内部可以划分为预约服务、设备服务、审批服务、通知适配、权限模块和数据访问层。C3 图不应重复 C2 的部署结构，而应解释内部职责。

【易错点】C1、C2、C3 的层次不能混用。把数据库、用户、Controller 和按钮画在同一张图里，会让图既看不清边界，也看不清职责。

## 5.5 UML 图：根据问题选择图

UML 不是必须全部使用的符号库。课程项目常用图包括用例图、类图、时序图、活动图和状态图。

用例图适合表达参与者和系统目标；类图适合表达领域对象及关系；时序图适合表达对象之间按时间顺序的交互；活动图适合表达流程分支；状态图适合表达对象生命周期。

在实验室预约系统中，“提交预约申请”可以用用例图说明参与者，用活动图说明流程分支，用时序图说明前端、后端、数据库和通知服务之间的调用，用状态图说明预约状态变化，用类图说明学生、设备、预约、审批记录之间的关系。

**UML 最小示例：提交预约申请**

| 图 | 最小内容 | 常见错误 |
|---|---|---|
| 用例图 | 学生、审批人、通知服务；提交预约、审批申请、发送通知 | 把页面按钮画成用例 |
| 时序图 | 前端调用后端，后端校验冲突，写数据库，生成通知任务 | 只画模块框，不画消息顺序 |
| 活动图 | 选择设备、校验冲突、提交、审批、通知，包含异常分支 | 把所有流程画成单线直走 |
| 状态图 | 待审批、已通过、已拒绝、已取消、已过期 | 把用户操作当成状态 |

【易错点】UML 图的价值不是符号完整，而是让业务规则、对象关系和状态变化可讨论。符号可以简化，关系不能含糊。

## 5.6 ERD 与 DFD：数据结构与数据流动

ERD（实体关系图）用于表达数据实体、属性和关系。实验室预约系统中的实体包括用户、角色、设备、实验室、预约、审批记录、通知记录和维护记录。ERD 可以帮助团队检查数据是否支持业务规则，例如一台设备在同一时间段不能有多个有效预约。

DFD（数据流图）用于表达数据在处理过程、外部实体和数据存储之间如何流动。它适合说明“数据从哪里来、经过哪些处理、存到哪里、输出给谁”。例如预约申请从学生输入进入系统，经过冲突校验和规则判断，生成预约记录和审批任务，再通知审批人。

ERD 关注“数据是什么结构”，DFD 关注“数据如何流动”。二者不能互相替代。

【考试提示】比较 ERD 和 DFD 时，应说明一个表达静态数据关系，一个表达动态数据流动，并结合具体业务说明。

## 5.7 状态与流程：不要把审批流程画成一条直线

预约业务不是简单的提交、审批、完成。真实系统需要处理待审批、已通过、已拒绝、已取消、已过期、设备维护中、通知失败等状态或异常。

状态图适合描述一个对象的生命周期。例如预约记录从“草稿”进入“待审批”，可能变成“已通过”或“已拒绝”；已通过后可能被取消；过期后进入历史状态。每个状态转换都应有触发事件和业务规则。

活动图或泳道图适合描述多个角色协作。例如学生提交，系统校验，教师审批，实验员确认设备状态，系统发送通知。泳道能清楚说明谁做动作、系统如何响应、规则在哪一步生效。

## 5.8 好图的检查清单

一张图是否有效，可以用以下清单检查。

| 检查项 | 判断问题 |
|---|---|
| 标题明确 | 这张图回答什么问题 |
| 层级一致 | 是否混合了不同抽象层 |
| 边界清楚 | 系统内外是否分明 |
| 关系清楚 | 连接线是否有方向和含义 |
| 术语一致 | 名称是否与需求和代码一致 |
| 可验证 | 能否推导测试或开发任务 |

图示不是越细越好。若一张图太复杂，应拆成不同层级或不同问题的图。若一张图无法帮助读者做判断，应删除或重画。

## 5.9 项目实践：提交一组最小建模图

课程项目建议提交以下最小图集。

| 图示 | 目的 |
|---|---|
| C1 上下文图 | 说明系统边界、用户和外部依赖 |
| C2 容器图 | 说明前端、后端、数据库和外部服务 |
| C3 核心模块图 | 说明后端主要职责划分 |
| 用例图 | 说明主要参与者和目标 |
| 类图/ERD | 说明核心数据对象关系 |
| 状态图或泳道图 | 说明关键流程和状态变化 |

每张图后面应配一段说明：这张图回答什么问题，哪些设计决策由它支撑，哪些测试可以从它推导。

**建模图质量评分表**

| 维度 | 合格标准 |
|---|---|
| 问题明确 | 图题能说明本图回答什么问题 |
| 层级一致 | 不把用户、按钮、数据库、类和服务混在同一层 |
| 边界清楚 | 系统内部、外部系统和用户角色有明确分界 |
| 关系可读 | 连接线有方向和含义，避免交叉混乱 |
| 术语一致 | 图中的名称与需求文档、代码模块、数据库表一致 |
| 可用于后续 | 能支撑开发任务、测试用例或架构决策 |

本章到下一章的衔接：本章解决“如何把系统结构表达清楚”，第6章将进一步讨论“为什么要这样组织结构、有哪些架构取舍、怎样记录架构决策”。

## 5.10 本章小结

1. 建模的目标是表达边界、关系、职责和变化。
2. C4 模型帮助团队按抽象层级表达系统结构。
3. UML 图应按问题选择，不能为了凑数量而画图。
4. ERD 表达数据结构，DFD 表达数据流动。
5. 好图必须能支撑沟通、开发、测试或评审。

## 5.11 复习题与参考答案要点

**基础概念题1：C4 模型的前三层分别是什么？**
参考答案要点：Context 系统上下文、Container 容器、Component 组件；分别表达系统环境、运行单元和内部职责。

**基础概念题2：ERD 和 DFD 的区别是什么？**
参考答案要点：ERD 表达实体、属性和关系；DFD 表达数据在外部实体、处理过程和数据存储之间的流动。

**基础概念题3：状态图适合表达什么问题？**
参考答案要点：适合表达对象生命周期、状态转换、触发事件和状态约束。

**简答题1：为什么说图示不是装饰？**
参考答案要点：图示应帮助说明边界、职责、数据、流程和状态，并支撑沟通、开发、测试和评审。

**简答题2：实验室预约系统至少需要哪些图？各自回答什么问题？**
参考答案要点：C1/C2/C3说明结构层级，用例图说明参与者目标，ERD说明数据关系，状态图或泳道图说明流程和状态变化。

**案例分析题：某系统图把用户、数据库、按钮、Controller、外部服务画在同一层。请指出问题并改进。**
参考答案要点：抽象层级混乱、边界不清、职责不明；应拆成C1上下文、C2容器、C3组件和必要的流程/状态图。

**项目实践题：为课程项目绘制一张 C2 容器图并说明外部依赖。**
参考答案要点：标出前端、后端、数据库、缓存/文件存储、外部认证/通知服务，并说明数据和调用关系。
