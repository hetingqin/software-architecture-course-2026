# 第1章 从能写代码到能控复杂系统

## 1.1 本章导读：为什么会写代码不等于能控复杂系统

软件系统最容易让初学者产生误解的地方在于：一个功能能运行，常常被误认为一个系统已经完成。对个人练习而言，这种判断有时可以成立；对多人协作、长期运行、涉及真实用户的数据系统而言，它远远不够。

以“实验室预约与设备管理系统”为例，最初的功能看起来并不复杂：学生提交预约，教师或实验员审批，系统记录设备状态，并发送通知。若只从代码实现角度看，可以很快写出页面、接口和数据库表。但系统一旦进入真实环境，就会遇到新的问题：两个学生同时预约同一台设备怎么办？通知服务失败是否影响预约成功？教师、实验员、学生看到的数据边界是否一致？学院以后要接入门禁、统一认证和设备台账，原来的代码是否还能扩展？

这些问题说明，软件工程关心的不只是“把功能写出来”，而是把功能放进一个可协作、可演进、可验证、可交付的工程系统中。软件体系结构课程的起点，也正是从“写代码”转向“控制复杂系统”。

【你要记住】功能能跑只是系统进入工程状态的最低条件。工程上真正要判断的是：系统是否能在变化、约束、协作和故障中保持可控。

## 1.2 一个反面案例：能跑但不可控的预约小系统

某项目组在两周内完成了实验室预约系统原型。学生可以选择设备和时间段，提交后数据库写入一条预约记录；教师可以在后台审批；预约成功后系统发送邮件。演示时一切顺利，页面也比较完整。

问题出现在试运行阶段。第一天，两个学生几乎同时预约同一台设备，系统生成了两条有效预约。第二天，邮件服务不稳定，提交预约的接口一直等待邮件发送结果，导致学生页面长时间转圈。第三天，学生通过修改浏览器请求参数访问了本不该看到的审批接口。第四天，学院提出后续要统计设备利用率，项目组发现原来的预约状态字段含义混乱，无法直接生成报表。

这个系统不是没有代码，也不是没有功能，而是没有把复杂性当成设计对象。它把“能提交表单”误认为“能支撑预约业务”，把“页面隐藏按钮”误认为“权限控制”，把“同步调用邮件接口”误认为“通知能力”，把“数据库能存”误认为“状态可分析”。

改进方向并不一定是使用更复杂的框架，而是先回答四个问题：需求会不会变化？质量属性是否明确？团队如何协作？运行和交付有什么约束？这四类问题构成了本课程理解复杂系统的基本入口。

## 1.3 软件复杂性的四个来源

软件复杂性通常不是单一来源造成的，而是多种力量叠加的结果。

第一类是需求规模与变化速度。实验室预约系统初版只做“预约”，后续可能增加设备借用、耗材管理、实验安排、门禁联动、学院统计。需求从一个表单扩展成一组业务流程，原来写死在页面或接口中的逻辑就会变成阻力。

第二类是团队协作规模。个人项目可以凭记忆维护代码；三到五人的项目必须划分职责；十人以上的项目需要统一接口、分支策略、评审机制和发布流程。协作规模越大，靠“大家沟通一下”解决问题的成本越高。

第三类是技术组合深度。一个系统可能同时涉及前端、后端、数据库、缓存、消息队列、身份认证、文件存储和第三方服务。技术组件越多，故障边界越难判断，版本兼容和数据一致性问题也越容易出现。

第四类是运行与交付约束。系统不是只在演示环境中运行，还要面对并发访问、网络抖动、数据备份、权限审计、日志追踪、版本升级和回滚。运行约束会把“代码问题”放大成“工程问题”。

【易错点】复杂性不是代码行数越多才出现。一个只有几千行代码的小系统，只要涉及真实用户、权限、并发、外部服务和长期维护，也会产生工程复杂性。

## 1.4 FR、QA、C：把问题分成三类

控制复杂系统的第一步，是把需求与约束分清楚。本课程使用一个简单但有效的三分法：功能需求、质量属性和约束条件。

功能需求（Functional Requirement, FR）回答“系统要做什么”。例如，学生提交预约、教师审批、实验员配置设备、系统生成预约记录。

质量属性（Quality Attribute, QA）回答“系统要做到什么程度”。例如，预约冲突不能发生，权限判断必须可靠，常用页面响应要足够快，通知失败不能拖垮主流程。

约束条件（Constraint, C）回答“系统必须在什么边界内完成”。例如，项目周期只有八周，团队只有四人，必须使用学校统一认证，部署环境只能使用学院服务器，期末答辩前必须交付可演示版本。

三者混在一起会导致方案失真。若把“不能重复预约”当成普通功能，就可能只在前端做一次判断；若把它识别为一致性质量要求，就会在数据库约束、事务边界和并发测试中体现出来。若忽略“统一认证”这个约束，后期再接入 CAS 或统一身份系统，改动会影响权限模型和接口设计。

## 1.5 架构时间账：今天省下的时间以后会不会还回去

软件开发中经常出现一种错觉：为了赶进度先写一个快版本，后面再优化。这个判断并非一定错误，但必须记录代价。否则短期节省的时间，会以调试、返工、上线故障和维护困难的形式还回来。

例如，预约冲突只靠前端判断，初期可以节省后端事务设计时间；但一旦出现并发冲突，就需要补数据库约束、修改接口、清理错误数据、重新测试。又如，权限只在页面隐藏按钮，初期看起来简单；但后续发现越权访问后，就要补统一鉴权、改所有接口、增加安全测试，代价远高于一开始设计清楚。

架构时间账不是要求所有事情一开始都做到完美，而是要求清楚记录：哪些地方是有意简化，为什么暂时接受，何时需要偿还，观察什么信号。软件工程中成熟的判断，不是“绝不欠债”，而是“不让无意识的债务滚成系统风险”。

【项目提示】课程项目可以接受简化设计，但必须写清楚简化理由。例如“通知先使用同步邮件发送”可以接受，但需要说明如果邮件失败如何处理、何时改成异步、如何避免主流程被拖慢。

## 1.6 软件工程的完整视角

一个软件系统从想法到交付，通常经历需求、设计、实现、测试、发布、运行和演进。每个阶段都可能产生工程判断。

需求阶段需要识别真实问题，而不是只收集“用户想要什么功能”。设计阶段需要确定结构、边界和关键质量属性。实现阶段需要把设计转化为代码和接口。测试阶段需要验证功能、质量和安全。发布阶段需要把系统可靠地交付到运行环境。运行阶段需要监控、日志、告警、故障处理和版本回滚。演进阶段需要在新需求和旧结构之间保持平衡。

本课程后续章节会沿着这个链条展开：过程模型决定开发节奏，项目管理决定资源与风险控制，需求工程决定问题是否定义清楚，建模与架构设计决定系统如何组织，测试和交付决定系统是否可验证、可上线、可维护。

【考试提示】解释“软件工程为什么不等于写代码”时，可以从需求、设计、协作、质量、交付和演进六个方面展开，并结合一个具体系统说明。

## 1.7 项目实践：给你的系统做一次复杂性体检

课程项目开始前，建议先完成一页“复杂性体检卡”。

| 检查项 | 可填写问题 | 示例 |
|---|---|---|
| 需求变化 | 后续最可能增加什么功能 | 设备统计、门禁联动 |
| 团队协作 | 哪些模块需要多人同时改 | 预约、审批、通知 |
| 技术组合 | 是否依赖外部服务 | 统一认证、邮件服务 |
| 运行交付 | 哪些故障会影响用户 | 通知失败、重复预约 |
| 质量属性 | 哪些指标必须保证 | 不重复预约、权限可靠 |
| 约束条件 | 时间、人力、部署边界 | 四人团队、八周交付 |

这个体检卡不要求一次性给出所有答案。它的作用是提醒项目组：写代码前要先判断复杂性来源，并把关键问题转化为后续需求、设计、测试和交付材料。

**实验室预约系统复杂性体检卡示例**

| 检查项 | 示例填写 | 后续影响 |
|---|---|---|
| 需求变化 | 后续可能增加门禁联动、设备统计、导师审批 | 第4章需求规格要保留扩展边界 |
| 团队协作 | 前端、后端、数据库、测试至少四类工作并行 | 第3章需要任务拆解和接口约定 |
| 技术组合 | 依赖统一认证、邮件服务、数据库事务 | 第6章需要明确外部依赖和一致性策略 |
| 运行交付 | 通知失败、重复预约、权限越权会影响真实使用 | 第7章、第8章需要测试和运行监控 |
| 质量属性 | 不重复预约、权限可靠、核心流程可追踪 | 后续架构、测试、交付都要围绕这些质量属性展开 |
| 约束条件 | 课程周期有限，团队经验有限，部署环境有限 | 第2章过程模型和第3章计划要控制首版范围 |

【考试提示】导论题常考“为什么软件工程不等于写代码”。答题时可以按“复杂性来源 → FR/QA/C 分类 → 架构时间账 → 后续工程活动”组织，不要只写“软件工程包括很多阶段”。

本章到后续章节的衔接：第2章将讨论怎样选择开发节奏来尽早暴露风险；第3章将把这些风险转化为计划、任务和责任人；第4章之后则逐步进入需求、建模、架构、测试和交付。

## 1.8 本章小结

1. 会写代码不等于能控制复杂系统，真实系统要面对需求变化、团队协作、技术组合和运行交付约束。
2. FR、QA、C 是分析软件问题的基础分类，能够帮助项目组区分“做什么”“做到什么程度”和“在什么条件下做”。
3. 架构时间账提醒项目组记录有意简化和未来偿还条件，避免无意识技术债。
4. 软件工程是从需求到运行演进的完整活动，代码只是其中一个环节。
5. 课程项目应从复杂性体检开始，而不是直接进入页面和接口开发。

## 1.9 复习题与参考答案要点

**基础概念题1：什么是软件复杂性的四个来源？**
参考答案要点：需求规模与变化速度、团队协作规模、技术组合深度、运行与交付约束。答题时应说明每一类复杂性如何影响项目，而不是只列名词。

**基础概念题2：FR、QA、C 分别表示什么？**
参考答案要点：FR 是功能需求，回答系统要做什么；QA 是质量属性，回答做到什么程度；C 是约束条件，回答必须在什么边界内完成。

**基础概念题3：什么是架构时间账？**
参考答案要点：架构时间账是对当前设计选择未来代价的记录和判断，包括简化原因、风险、偿还条件和观察指标。

**简答题1：为什么“功能能跑”不代表系统工程上已经完成？**
参考答案要点：功能能跑只说明局部流程可执行；工程完成还需要考虑并发、权限、可靠性、可维护性、可测试性、部署运行和后续演进。

**简答题2：用实验室预约系统说明 FR、QA、C 的区别。**
参考答案要点：提交预约是 FR；不能重复预约、权限可靠、响应时间可接受是 QA；必须接入统一认证、八周内交付、部署在学院服务器是 C。

**案例分析题：某学生项目把所有权限控制都放在前端页面隐藏按钮中。请分析风险并提出改进方案。**
参考答案要点：风险包括越权访问、接口被直接调用、审计困难；改进方案包括后端统一鉴权、角色权限模型、接口级安全测试、日志记录和权限异常告警。

**项目实践题：为你的课程项目填写一张复杂性体检卡。**
参考答案要点：至少覆盖需求变化、团队协作、技术组合、运行交付、质量属性和约束条件，并说明最优先处理的两个风险。
