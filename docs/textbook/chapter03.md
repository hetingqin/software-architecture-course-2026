# 第3章 项目管理、可行性分析与风险控制

## 3.1 本章导读：项目管理不是填表，而是让目标可执行

软件项目失败时，问题往往不只出在技术。目标没有对齐、范围不断膨胀、任务无人负责、进度不可见、风险没有提前处理，都会让一个看似简单的系统变得难以收尾。

实验室预约与设备管理系统可以被写成一句话：“做一个预约系统”。但这句话无法指导项目执行。项目组需要进一步回答：服务哪些用户？首版交付哪些功能？哪些功能不做？哪些风险先验证？任务如何拆分？什么时候集成？如果统一认证接入失败怎么办？如果团队成员进度不一致怎么办？

项目管理的价值，是把模糊目标转化为可执行、可跟踪、可调整的工程活动。本章关注可行性分析、项目计划、任务拆解、风险控制、沟通机制和工具使用。

【你要记住】项目管理不是额外负担，而是让团队知道“现在做什么、谁负责、何时交付、风险在哪里”的工程机制。

## 3.2 一个反面案例：没有计划的“最后一周集成”

某项目组四名成员分工明确：一人做前端，一人做后端，一人做数据库，一人做文档。前几周每个人都在“推进自己的部分”，但没有统一接口约定，也没有阶段集成。到最后一周，前端字段名和后端接口不一致，数据库状态枚举和页面显示不匹配，通知功能没有测试环境，项目文档也无法反映真实实现。

更麻烦的是，团队没有风险清单。统一认证接口能否接入、预约冲突如何处理、文件上传是否需要权限限制、统计报表是否要导出，这些问题都在临近答辩时集中出现。项目不是没有人做事，而是缺少把工作组织起来的机制。

改进方式不是增加复杂管理工具，而是尽早建立四类基本产物：范围边界、里程碑计划、任务看板、风险清单。它们让团队能及时发现“哪些工作没有主人”“哪些假设还没有验证”“哪些接口必须提前对齐”。

## 3.3 可行性分析：先判断做不做得成

可行性分析不是大型项目才需要。课程项目也应在开工前判断技术、时间、资源和业务可行性。

技术可行性关注团队是否具备实现关键功能的能力。例如，是否会处理并发预约冲突，是否能接入统一认证，是否能部署后端服务。时间可行性关注任务规模是否匹配学期周期。资源可行性关注人力、测试数据、服务器和外部接口是否可获得。业务可行性关注系统是否真正解决目标用户问题。

对实验室预约系统而言，首版可行方案应避免把所有设想一次做完。可以先保证预约、审批、状态流转、基础权限和数据记录；把门禁联动、复杂统计、设备传感器接入放入后续版本或扩展目标。

【易错点】可行性分析不是证明“项目一定能做”，而是识别哪些目标必须缩小、哪些风险必须提前验证。

## 3.4 范围管理：先说清楚不做什么

范围管理的关键不只是列出要做功能，还要明确首版不做什么。对课程项目来说，范围越模糊，越容易出现“每个人都觉得应该加一点”的范围蔓延。

例如，实验室预约系统首版可以定义为：学生预约、教师审批、实验员管理设备、系统发送基础通知、管理员查看预约记录。暂不做移动端 App、门禁硬件联动、复杂推荐算法、跨学院共享、自动排课。这个边界不是降低要求，而是保证核心业务闭环可交付。

范围边界要与验收标准绑定。若写“设备管理”，应说明包括设备新增、停用、维护状态和可预约状态；若写“通知”，应说明通知触发条件、失败处理和是否影响主流程。

【项目提示】项目计划中建议加入“本版本不做清单”。它能减少后期争论，也能体现团队对可交付范围的控制能力。

## 3.5 WBS 与里程碑：把大目标拆成可交付单元

WBS（工作分解结构）用于把项目目标拆成较小的工作包。好的工作包应具备清晰产出、明确负责人、可估算工作量、可验收标准。

实验室预约系统可以拆成：用户与权限、设备管理、预约申请、审批流、冲突校验、通知服务、统计报表、部署与测试、项目文档。每个工作包再拆为接口、页面、数据表、测试用例和文档。

里程碑不是“第几周写代码”这种模糊描述，而是可验证状态。例如：第2周完成预约流程原型评审；第4周完成预约闭环可演示；第6周完成权限与通知集成；第8周完成测试、部署和答辩材料。

| 工作包 | 子任务 | 负责人 | 产出物 | 验收方式 |
|---|---|---|---|---|
| 预约申请 | 预约表单、冲突校验、提交接口 | 后端A/前端B | 页面、接口、测试用例 | 能提交待审批预约，冲突时给出提示 |
| 审批流程 | 审批列表、审批详情、状态流转 | 后端C/前端B | 审批页面、状态机说明 | 审批通过/拒绝后状态正确 |
| 设备管理 | 设备新增、停用、维护状态 | 后端A | 数据表、管理页面 | 维护中设备不可预约 |
| 通知服务 | 通知任务、失败记录、重试说明 | 后端C | 通知模块、日志 | 邮件失败不影响预约保存 |
| 测试与部署 | 测试数据、部署说明、演示环境 | 全组 | 测试报告、部署文档 | 从零部署可运行 |

**八周里程碑样例**

| 周次 | 里程碑 | 可验收成果 |
|---|---|---|
| 第2周 | 原型与需求评审 | 预约主流程原型、核心用例、范围边界 |
| 第4周 | 预约闭环 | 学生提交、审批人审批、状态可查询 |
| 第6周 | 权限、通知与测试 | 角色权限、通知失败处理、核心测试用例 |
| 第8周 | 部署与答辩 | 可运行系统、部署说明、项目文档、演示脚本 |

【项目提示】WBS 不要按“前端、后端、数据库”粗分到底，而要落实到可验收的业务成果。否则每个人都在做事，但系统闭环迟迟无法出现。

## 3.6 风险管理：把“可能出事”写成可跟踪事项

风险管理不等于写几句“时间不够、技术难度大”。有效风险项至少包括：风险描述、触发条件、影响范围、概率、影响程度、应对措施、负责人和观察信号。

实验室预约系统常见风险包括：并发预约导致冲突；统一认证接口接入失败；通知服务不可用；审批规则临时变化；团队成员接口约定不一致；测试数据不足。每个风险都应对应具体动作。例如“统一认证接入失败”的应对措施可以是先定义认证适配接口，准备本地账号模拟方案，并在第3周前完成接口验证。

风险矩阵可以把概率和影响结合起来。高概率高影响风险必须提前处理；低概率低影响风险可以记录观察；高影响但概率不确定的风险应做技术验证。

## 3.7 沟通与会议：让信息流动，而不是制造形式

软件项目中的会议只有在形成决策、暴露阻塞、同步状态或沉淀复盘时才有价值。常见会议包括立项会、需求评审、方案评审、每日站会、版本评审和复盘会。

立项会要对齐目标、范围和角色；需求评审要确认用户、场景和验收标准；方案评审要暴露架构风险和放弃方案；站会要解决阻塞而不是汇报流水账；复盘会要记录做得好的地方、出现的问题和下一轮改进。

对课程项目而言，推荐使用轻量机制：每周一次版本会，记录本周完成、下周计划、风险变化和待决策问题；关键设计变更必须形成记录，不要只停留在聊天记录里。

## 3.8 工具链：工具服务于透明度

工具不是项目管理的核心，但合适工具能提高透明度。任务看板用于展示待做、进行中、待评审、已完成；Issue 或 Ticket 用于记录任务、缺陷和需求变更；Pull Request 用于代码评审和合并；CI 用于自动构建和测试。

一个小团队可以使用 Git 仓库、看板、文档协作工具和简单的版本记录。重要的不是工具名称，而是信息是否可追踪：任务是否有负责人，接口变更是否有记录，缺陷是否能复现，版本是否能回退。

【考试提示】解释项目管理工具作用时，不要只列工具名，应说明它们如何支持任务可见、协作可追踪、质量可验证和发布可控制。

**轻量 Ticket 与 PR 样例**

| 类型 | 示例 |
|---|---|
| Ticket 标题 | 增加预约提交接口的冲突校验 |
| Ticket 内容 | 背景：当前接口只保存预约；要求：同一设备同一时段只能存在一条有效预约；验收：并发提交时只能成功一条，其余返回冲突提示 |
| PR 检查项 | 是否更新接口说明；是否补充冲突测试；是否影响审批状态；是否有数据库迁移脚本 |
| CI 检查项 | 后端测试通过；前端构建通过；数据库初始化脚本可执行 |

【考试提示】项目管理题如果问“工具如何支持协作”，答题应落到任务可追踪、接口可评审、代码可合并、质量可验证，而不是只列工具名称。

## 3.9 项目计划样例：实验室预约系统

一个简化项目计划可以包含以下元素。

| 元素 | 示例 |
|---|---|
| 项目目标 | 完成预约、审批、设备管理、通知和基础统计 |
| 首版范围 | Web端，支持三类角色，完成预约闭环 |
| 不做范围 | 门禁硬件联动、移动App、复杂推荐 |
| 里程碑 | 原型评审、预约闭环、权限通知集成、部署验收 |
| 风险 | 并发冲突、统一认证、通知失败、团队接口不一致 |
| 质量要求 | 不重复预约、权限可靠、核心流程可测试 |
| 交付物 | PPT、需求说明、架构图、测试报告、部署说明 |

这类计划不追求复杂，但必须能让项目组回答：现在处在哪个阶段，下一步交付什么，哪些风险还没有关闭。

本章到下一章的衔接：项目计划解决“怎样组织开发”，但计划的输入必须是清晰需求。第4章将讨论如何把用户声音转化为可验证规格。

## 3.10 本章小结

1. 项目管理的核心是让目标、范围、任务、风险和交付状态可见。
2. 可行性分析帮助团队提前判断哪些目标应收缩，哪些风险应验证。
3. WBS 和里程碑把大目标拆成可交付单元，避免最后一周集成。
4. 风险管理要写成可跟踪事项，而不是泛泛而谈。
5. 工具链的价值在于提高透明度和可追踪性。

## 3.11 复习题与参考答案要点

**基础概念题1：什么是可行性分析？**
参考答案要点：从技术、时间、资源、业务等方面判断项目目标是否可实现，并识别需要缩小或提前验证的风险。

**基础概念题2：WBS 的作用是什么？**
参考答案要点：把项目目标拆分为可负责人、可估算、可验收的工作包，支持计划、协作和跟踪。

**基础概念题3：风险项应包含哪些信息？**
参考答案要点：风险描述、触发条件、影响范围、概率、影响程度、应对措施、负责人和观察信号。

**简答题1：为什么项目计划中要写“不做范围”？**
参考答案要点：防止范围蔓延，明确首版交付边界，减少团队争议，保证核心闭环按时完成。

**简答题2：说明任务看板、Ticket、PR、CI 分别如何支持项目管理。**
参考答案要点：看板显示状态，Ticket 记录任务和缺陷，PR 支持评审和合并，CI 自动构建测试并暴露质量问题。

**案例分析题：某团队最后一周才发现前后端接口不一致。请从项目管理角度分析原因和改进。**
参考答案要点：原因包括缺少接口评审、里程碑集成、任务跟踪和风险管理；改进包括接口契约、早期集成、版本评审、CI 和任务责任人。

**项目实践题：为课程项目编写一页风险清单。**
参考答案要点：至少列出5项风险，每项包含影响、触发条件、应对措施、负责人和关闭时间。
