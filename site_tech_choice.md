# 学生端课程网站技术选型

## 一、候选方案

本课程网站以 Markdown 文档、PDF 下载和课程资料导航为主，不需要复杂交互。对比三种方案：

1. MkDocs Material；
2. Docusaurus；
3. Astro。

| 视觉定制 | 足够 | 较强 | 最强 |

## 三、方案分析

### 3.1 MkDocs Material

优势：

- Markdown-first，最适合课程文档站；
- 导航、搜索、目录、代码块、提示框等能力成熟；
- 教师后续维护成本低；
- 构建速度快；
- 静态部署简单；
- 不需要写复杂前端代码；
- 适合“教材 + 课件 + 作业 + 复习说明”的资料型网站。

不足：

- 高度视觉定制不如 Astro；
- 如果未来要做复杂交互，扩展空间不如前端框架灵活。

判断：最适合当前课程网站。

### 3.2 Docusaurus

优势：

- React 生态，适合大型文档站；
- 版本管理、博客、文档导航能力较强；
- 主题成熟；
- 对后续扩展交互组件比较友好。

不足：

- Node/React 工程复杂度高于 MkDocs；
- 对本课程当前目标来说略重；
- 教师后续维护需要理解前端项目结构。

判断：如果未来课程网站要发展为大型课程资源平台，可以考虑；当前阶段不优先。

### 3.3 Astro

优势：

- 视觉表达能力强；
- 静态站点性能好；
- 可以做更精致的课程门户；
- 适合自定义首页和复杂视觉设计。

不足：

- 需要更多前端页面组织；
- 文档站能力需要额外搭配内容集合、搜索和导航方案；
- 对当前资料发布任务来说开发成本偏高。

判断：适合后续做课程品牌展示站，不适合当前“快速上线、便于维护”的第一版。

## 四、推荐方案

推荐采用：**MkDocs Material**。

推荐理由：

1. 当前网站核心是资料发布，不是复杂 Web 应用；
2. Markdown 管理方式最适合课程资料持续更新；
3. 学生查阅体验稳定，搜索和导航能力足够；
4. 教师后续只需维护 Markdown 和 PDF；
5. 可快速部署到 GitHub Pages、Gitee Pages 或学校服务器；
6. 便于加入发布前保密校验脚本。

## 五、建议技术栈

```text
Python 3.x
MkDocs
mkdocs-material
Markdown
静态资源目录 site/docs/assets/
保密校验脚本 scripts/validate_public_assets.py
```

## 七、部署方式

### 7.1 GitHub Pages

适合公开课程资源；如果课程资料只面向校内，需注意公开范围。

### 7.2 Gitee Pages

国内访问速度较好，适合学生下载 PDF。

### 7.3 学校服务器

适合校内课程资料发布，可由信息化或学院服务器提供静态目录。

### 7.4 本地静态包

可将构建后的 `site/` 静态文件打包，上传到学校课程平台。

## 八、保密发布机制建议

无论采用哪种技术，发布前必须运行保密扫描：

1. 扫描 `site/docs/` 与 `site/public/`；
2. 检查文件名敏感关键词；
3. 检查是否误复制 `exam/` 文件；
4. 检查是否包含 `teacher_only`、`答案`、`评分`、`试卷` 等内容；
5. 检查站点链接是否指向上级目录中的保密文件。

构建流程建议：

```text
整理公开资源 → 复制白名单文件 → 生成 Markdown 页面 → 保密扫描 → 构建站点 → 人工抽查 → 发布
```

## 九、结论

第一版学生端课程网站建议采用 MkDocs Material。它能以最小工程复杂度完成课程资料发布、教材下载、逐讲导航、作业说明和复习说明，并且便于加入保密校验机制。

Docusaurus 和 Astro 暂不作为第一版首选。若后续需要更强交互或更复杂视觉展示，可在 MkDocs 版本稳定后再评估升级。
